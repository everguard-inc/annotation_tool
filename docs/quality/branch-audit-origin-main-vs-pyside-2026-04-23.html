Bottom Line

origin/pyside is not yet a trustworthy statement of parity with origin/main. It already contains the large PySide port for project shell, labeling, filtering, completion, and update workflows, but several baseline features are either missing, only partially hardened, or missing runtime assets needed to work in practice.

The current worktree is materially ahead of the remote PySide branch. It closes the concrete gaps that matter most for the approved Linux first-release scope: SQLite-to-JSON migration on first open, event-validation PySide parity, status-bar adaptive font behavior, help/hotkeys template bundling, invalid-data_dir recovery, overwrite refresh behavior, and a much broader proof surface in tests.

Even in the current worktree, this is still not literal 1:1 Tk parity. The explicit platform non-parity decision is macOS, intentionally held on the Tk line for the first PySide release. There is also a release-process caveat: the current docs still classify 9 of 45 audit-tracked test files as rewrite-or-augment or review-before-accept, so full production promotion still depends on the remaining rehearsal and human parity checks.

Audit Method

  • Inspected git refs and commit history for origin/main, origin/pyside, and the checked-out worktree.
  • Used docs/product/workbook-crosswalk.md as the normalized origin/main feature inventory (197 workbook FR rows).
  • Compared representative legacy Tk code in origin/main:main.py to PySide implementations in annotation_tool/ui/main_window.py, annotation_tool/ui/actions.py, annotation_tool/services/import_export_service.py, annotation_tool/services/session_state.py, and the repositories/widgets layer.
  • Checked test coverage shape from docs/quality/pyside-test-disposition.md and extracted distinct FR citations directly from test files on both refs.
  • Ran the repository’s required verification lane on the current worktree.

Current-State Assessment

If the question is "does the current worktree cover the approved PySide first-release scope well enough to be taken seriously as release work?", the answer is yes. The current tree passes the repository’s standard pytest lane, cites all 197 FR rows in tests, and closes the concrete branch gaps that were still visible on origin/pyside.

If the question is "does the current worktree fully replace origin/main without qualification?", the answer is no. The project is still explicitly non-parity for macOS. That is an accepted release-scope decision rather than an accidental miss.

The remaining uncertainty is therefore mostly operational rather than architectural: human Tk-vs-PySide visual parity review, rollback confirmation after a PySide write in a safe scenario, and backend-sensitive acceptance for fresh install and in-app update.

Architecture Change Diagram

The biggest structural change was not just Tk to PySide. The application moved from a window-centered orchestration model to a layered desktop architecture, where the window composes services and screens instead of owning most of the workflow logic directly.

Before: origin/main Tk architecture

MainWindow as orchestrator Menu wiring, project open/download/remove/complete, update flow, help windows, shutdown behavior.
↓ direct calls into everything
Widget factory and GUI helpers annotation_widgets.factory, gui_utils, dialogs, HTML windows, form helpers.
Procedural integrations api_requests, path_manager, config, utils, direct subprocess and filesystem operations.
Legacy local state state.json plus Tk-era db.sqlite; widget/runtime logic was closely coupled to the storage shape.

After: current PySide layered architecture

ui/ as shell and presentation MainWindow, dialogs, screens, widgets, actions, exception surfacing.
↓ composes and delegates
services/ as workflow orchestration ProjectService, CompletionService, ImportExportService, per-mode sessions, runtime state handling.
↓ uses explicit boundaries
infrastructure/ as IO boundary API client, file transfer, unzip, DB wrapper, repositories for project/filtering/labeling persistence.
annotation/ and media/ as domain/runtime engines Figure controllers, bbox/keypoint/mask logic, drawing, video frames, image conversion.
core/ plus JSON stores Shared enums/models/settings/paths/exceptions, with PySide-owned cache.json and runtime_state.json.
Tk baseline
UI/widget logic had a shorter path to backend, filesystem, and SQLite-era project storage.
Current PySide runtime flow
screen/widget → session/service → repository/client → JSON store / backend
Compatibility bridge
First open can migrate Tk db.sqlite data into PySide cache.json and runtime_state.json.
