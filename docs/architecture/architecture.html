PySide Annotation Tool Architecture

The app is a Qt desktop shell that mounts one project workflow at a time. Screens handle input, sessions own workflow state, repositories own local persistence, and import/export services translate backend artifacts into local caches.

Shell UI Services Repositories Annotation domain

Layered Design

Shell

MainWindow loads settings, builds shared services, routes menu actions, and chooses the active screen.

  • ui/main_window.py
  • ui/actions.py

Screens and Widgets

Workflow screens translate Qt events into session calls. Widgets render canvas, status bars, classes, and HTML.

  • ui/screens/
  • ui/widgets/

Services

Sessions own item navigation, runtime state, timing, status data, and workflow commands.

  • services/*_session.py
  • services/import_export_service.py

Repositories

Repositories define cache schemas, load/save workflow data, and migrate from Tk-era SQLite when needed.

  • infrastructure/repositories/
  • core/paths.py

Annotation Domain

Figure models and FigureController implement drawing, hit testing, history, clipboard, masks, and keypoints.

  • annotation/
  • media/

Shared Input Pipeline

Qt Event

Canvas, keyboard, combo box, text edit, slider, or menu action.

Screen

Thin handler maps UI input to a session command.

Session

Updates workflow state, tracks action time, saves current item when navigating.

Repository

Reads or writes cache.json and performs migration fallback.

Refresh

Screen redraws canvas and updates status/sidebar controls.

Workflow Details

Annotation

Stage controls the item list and label catalog. Non-review stages edit figures. Review stage places review-label callouts.

Load image annotations from cache.json.
Create FigureController for project mode or review labels.
Mouse and keyboard events mutate controller figures.
Render image with overlays, blur regions, review labels, and status.
Navigation saves serialized figures or review labels.

Key files: labeling_screen.py, labeling_session.py, labeling_repository.py, annotation/.

Filtering

Each video frame is an item. A barcode decoder supplies a stable image name when possible; otherwise the frame id remains valid.

Open video.mp4 with VideoFrameProvider.
Request current frame and decode item name.
Toggle selected state in FilteringRepository.
Render selected frames with a green border.
Export selected names or fallback ids to selected_frames.json.

Key files: filtering_screen.py, filtering_session.py, filtering_repository.py, media/.

Event Validation

Each event uid maps to media plus ordered answers. The sidebar edits answers/comments while the canvas shows video or image evidence.

Build fields from meta.json and events from media/results.
Load video frames, capped by MAX_EVENT_VALIDATION_VIDEO_FRAMES.
Sidebar combo boxes update ordered answers.
Keyboard and playback controls step frames/events.
Export ordered answers and comments to event_validation_results.json.

Key files: event_validation_screen.py, event_validation_session.py, event_validation_repository.py.

Persistence and Artifact Boundaries

File Owner Purpose
state.json ProjectRepository Local project id, uid, stage, and mode.
runtime_state.json SessionStateStore Current item, duration hours, and processed item ids.
cache.json Mode repository Working cache for labels, items, figures, selections, or event answers.
figures.json, review.json ImportExportService Labeling artifacts downloaded from and uploaded to the backend file service.
selected_frames.json FilteringScreen or ImportExportService Completion export for filtering selections.
event_validation_results.json EventValidationScreen or ImportExportService Completion export for event answers and comments.

Design Guardrails

  • Keep screens thin. Workflow rules belong in sessions, repositories, or services.
  • Keep cache schema knowledge inside repositories and import/export.
  • Use core/paths.py for persisted paths.
  • Do not let overwrite save stale screen state before reloading downloaded annotations.
  • For event validation, preserve field order because answers are stored as ordered lists.
Current completion gap: EventValidationRepository.count_incomplete() can detect unanswered events, but CompletionService.check_before_completion() currently returns success for all workflows. Wire this deliberately if completion should block incomplete validation.
